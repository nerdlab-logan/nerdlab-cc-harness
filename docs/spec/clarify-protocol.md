# Clarify 프로토콜

`/nl-plan` 의 첫 단계. **메인 세션** 이 수행한다 (헤드리스 격리 원칙의 명시적 예외 — 이유는 [philosophy.md](../philosophy.md) 의 "Clarify 예외" 섹션 참조).

이 단계의 목적: 사용자 요구사항이 plan 으로 구체화될 만큼 충분한지 게이트하고, 부족분을 사용자와 한 번에 묶어 보충한다. clarify 는 "단계" 보다 "게이트" 에 가깝다 — 이미 정리된 입력이면 거의 통과만 하고, 큰 그림만 있으면 질문 라운드로 보충한다.

## 책임

- ★ 항목 추출 (이번 세션 대화 + `/nl-plan` 인자)
- 항목별 상태 판정 (`filled` / `empty` / `ambiguous`)
- 빈/모호 항목을 한 라운드에 묶어 질문 (`AskUserQuestion` 우선, 자유 서술은 산문)
- 모든 항목 확정 후 사용자에게 요약 보여주고 진행 확인
- 확정된 ★ 요약을 다음 단계 (context gather → planner) 로 인계

## 입력 ★ 항목 (6개)

clarify 가 추출/검증하는 항목. plan-template 의 ★ 섹션(출력 차원)과는 다른 **입력 차원**의 항목이다. 이 6개가 채워지면 planner 가 plan-template 의 10개 섹션을 자신 있게 채울 수 있다.

| # | 항목 | 정의 | 빈(empty) 판정 | 모호(ambiguous) 판정 |
|---|------|------|---------------|---------------------|
| 1 | **목적** | 이 작업이 끝났을 때 달성될 상태 | 언급 자체 없음 | "잘 작동", "문제없이" 등 검증 불가 어휘만 |
| 2 | **입력** | 들어오는 데이터·인자·사용자 액션 | 입력 형태 언급 없음 | 형식/스키마/출처 미정 (예: "데이터 받아서") |
| 3 | **출력** | 만들어지는 결과·파일·UI·상태 변화 | 출력 형태 언급 없음 | 형태 미정 (예: "결과 보여줘") |
| 4 | **제약** | 기술 스택·성능·호환·보안 조건 | 언급 없으나 작업 성격상 명백히 필요 | "빠르게", "안전하게" 같은 정량 불가 |
| 5 | **완료조건** | 어떻게 끝난 줄 아는가 (테스트·수동 확인) | 언급 없음 | "잘 되면 끝" 류, 검증 가능 명령/관찰 부재 |
| 6 | **안 할 것** | 의식적으로 제외하는 인접 영역 | 언급 없음 | — (이 항목은 모호 판정 안 함, 비면 메인이 휴리스틱으로 1~3개 제안 후 사용자 확인) |

## 추출 절차

1. **컨텍스트 수집** — 이번 세션의 대화 전체 + `/nl-plan` 인자 (있으면).
2. **항목별 추출 시도** — 6개 항목 각각에 대해 텍스트 후보를 뽑는다.
3. **항목별 상태 판정** — 위 표의 빈/모호 기준으로 `filled` / `empty` / `ambiguous` 분류.
4. 판정 자체가 어려우면 자문: "이 항목만으로 plan-template 의 해당 섹션을 자신 있게 채울 수 있나?" → 자신 없으면 `ambiguous`.

## 질문 절차

**라운드 1** (빈/모호 항목 ≥ 1):
- 모든 빈/모호 항목을 **한 번에 묶어** 질문. 한 항목씩 N번 묻지 X.
- `AskUserQuestion` 도구를 우선 사용:
  - 객관식이 가능한 항목 (예: 출력 형태 = JSON / CSV / DB / UI / 기타) 은 `options` 제공
  - 자유 서술 필요한 항목 (목적·제약 등) 은 `multiSelect: false` + `options` 마지막에 "기타(서술)" 두기
- 한 호출에 question N개 (각 항목 1개씩) 동시 전달.
- 추가 맥락 필요한 자유 서술은 산문 보조 질문 1회로 묶어 보냄.

**라운드 2** (라운드 1 답변 후 여전히 ambiguous ≥ 1):
- 같은 방식, **잔여 항목만**.

**라운드 2 초과** → `abort` 옵션을 사용자에게 위임:
- (가) 미명시 항목은 메인이 자율 판단으로 가정값 박고 진행 — 가정은 planner 입력에 명시되며 plan 의 "트레이드오프" 또는 "안 할 것" 에 흡수
- (나) 작업 취소 (`/nl-plan` 종료, 사용자가 요구사항 보강 후 재호출)

## 추출 결과 투명화

모든 ★ 항목이 확정된 후 (시나리오 A/B/C 무관) **항상** 한 번 사용자에게 요약 표시 + 확인:

```
clarify 결과 요약:
- 목적: <한 줄>
- 입력: <한 줄>
- 출력: <한 줄>
- 제약: <한 줄>
- 완료조건: <한 줄>
- 안 할 것: <항목1>, <항목2>

이대로 진행할까요? (수정할 항목이 있으면 말씀해주세요)
```

사용자가 "X 항목만 수정" 답하면 → 그 항목 갱신 → 재요약 → 재확인 (이 왕복도 라운드로 카운트, 최대 2).

투명화 단계는 스킵 불가. 시나리오 A (대화로 이미 정리됨) 라도 메인 추출이 잘못됐을 수 있으므로.

## 산출물

확정된 ★ 요약 **텍스트 한 덩어리** — `/nl-plan` 의 다음 단계 (context gather → planner 헤드리스 호출) 의 user prompt 로 전달된다.

별도 파일 저장 X. plan.md 의 다음 섹션으로 자연스럽게 흡수된다:
- 목적 → `1. 목표`
- 입력/출력/제약 → `8. 데이터 흐름`, `시그니처` 결정 근거
- 완료조건 → `9. 검증 방법`
- 안 할 것 → `3. 이번에 안 할 것 ★`

## 라운드 카운팅 룰

- 라운드 = "메인이 사용자에게 질문 보낸 횟수" (요약 확인의 수정 왕복도 1회 카운트)
- 최대 2 라운드 누적 시 abort 옵션 자동 제시
- 라운드 0 (질문 없음, 모두 filled) 케이스도 정상 — 요약 확인은 라운드로 안 셈

## 안티패턴

- 빈/모호 항목 발견했는데 메인이 자체 판단으로 채우고 진행 — abort 옵션 거치지 않으면 금지
- 한 항목씩 한 번에 한 질문 보내기 — 라운드 한도 빠르게 소진
- 추출 요약 확인 단계 스킵 (시나리오 A 라도 필수)
- `prompts/clarifier.md` 같은 헤드리스 clarifier 호출 — 세션 맥락 손실
- `tasks/<name>/clarify.md` 별도 파일 생성 — plan.md 와 동기화 부담
- 사용자 답변 받았는데 라벨/형식만 확인하고 의미는 그대로 두기 (재판정 필수)
