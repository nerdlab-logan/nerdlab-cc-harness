# Plan 출력 템플릿

`planner` 가 출력하는 모든 plan 파일은 이 형식을 따른다. `nl-generate` / `nl-review` 가 plan 을 이 구조로 가정해 소비한다. 이 파일이 단일 진실의 출처(single source of truth)이며, 형식을 바꿀 때는 여기만 수정한다.

★ 의 두 차원을 혼동하지 마라:
- **출력 ★** (이 파일) — plan 의 어느 섹션이 비면 reviewer 가 plan 자체를 거부 (3·4·10·11·12번)
- **입력 ★** ([clarify-protocol.md](clarify-protocol.md)) — 사용자 요구사항 차원. 이 6개가 채워져야 planner 가 본 템플릿을 채울 수 있음. clarify 단계에서 게이트.

## 저장 경로

`tasks/<task-name>/plan.md` — task-name 은 요구사항을 한 줄로 압축한 kebab-case (예: `add-login-rate-limit`).

같은 디렉토리에 `nl-generate` 실행 시 다음 파일이 함께 생성된다 (`scripts/run_phases.py` 가 관리):
- `status.json` — 전체 진척
- `phase{N}.status.json` — phase 별 상태
- `phase{N}-round{M}.log` — round 별 로그

## 필수 섹션 (순서 고정)

1. **목표** — 한 줄. 이 plan 이 끝났을 때 달성되어야 할 상태.
2. **핵심 변경** — 만들 것. 번호 매긴 리스트, 각 항목은 동사로 시작.
3. **이번에 안 할 것** ★ — out of scope 항목. **비울 수 없음 (≥1)**. scope creep 1차 방어선.
4. **트레이드오프** ★ — 표 (`결정` / `이유` / `포기한 것`). **≥1행**. reviewer 가 나중에 동일 영역 흔들지 않게.
5. **TDD** — `yes` 또는 `no` + 이유 한 줄.
6. **변경 파일** — `[신규|수정|삭제] 경로 — 변경 요약` 한 줄씩.
7. **시그니처** — 신규 함수/타입 시그니처 코드블록 (본문은 비움 — 구현은 coder 몫).
8. **데이터 흐름** — 입력 → 변환 → 출력. 3~5줄.
9. **검증 방법** — 실행할 명령어 + 사람 눈 확인 항목 (UI 변경 시).
10. **Phase 분해** ★ — 표 (`번호` / `이름` / `범위 한 줄` / `의존`). 작은 작업도 `Phase 1: 단일` 한 줄로 표기. `nl-generate` 가 이 섹션을 파싱해 직렬 실행.
11. **Context** ★ — Phase A/B Context Gather 결과. `### 영향 파일 ★` / `### 관련 시그니처 ★` / `### 기존 패턴 ★` 세 하위 섹션 필수. 빈 코드 저장소(`.git` 없음 또는 추적 파일 0개)이면 `none (empty codebase)` 한 줄로 대체 가능.
12. **Evaluate** ★ — phase 통과 직후 자동 실행할 외부 검증 명령. 셸 명령을 줄 단위로 나열. 실행할 명령이 없으면 `none` 한 줄로 명시 (빈 칸 금지).

★ 표시: 비어 있으면 reviewer 가 plan 자체를 거부할 수 있는 항목.

### ★ 섹션 빈/placeholder 식별 패턴

`/nl-plan` 종료 시 메인이 정규식으로 검증하고, reviewer 1차 호출 시 항목별 status 도 강제한다. "비었다"의 정의:

| 섹션 | 비었다고 판정하는 패턴 |
|------|----------------------|
| 3. 이번에 안 할 것 | 항목 0개 / `TBD` / `없음` / `-` 한 줄 / placeholder 어휘만 |
| 4. 트레이드오프 | 표 헤더만 있고 데이터 행 0개 / 한 행이 모두 `-` |
| 10. Phase 분해 | 표 헤더만 있고 데이터 행 0개 / 범위 칸이 비거나 `TBD` |
| 11. Context | `### 영향 파일 ★` / `### 관련 시그니처 ★` / `### 기존 패턴 ★` 세 헤더 중 하나라도 없음 / placeholder 어휘만 / 완전히 비어 있음. 단, `none (empty codebase)` 한 줄은 합법. |
| 12. Evaluate | 명령 줄 0개 / `TBD` / `없음` / `-` 한 줄 / placeholder 어휘만. 단, `none` 한 줄은 합법 (실행할 명령 없음 명시). |

## TDD 판단 기준

| 작업 성격 | 권장 |
|-----------|------|
| 비즈니스 로직 · 파서 · 계산 · 인증/권한 | yes |
| UI 시안 · 일회성 스크립트 · CSS · 문서 · 마이그레이션 | no |
| 사용자가 `/nl-generate --tdd` 또는 `/nl-plan --tdd` 명시 | 무조건 yes |

## Phase 분해 가이드

| 작업 크기 | phase 수 | 예시 |
|-----------|---------|------|
| 단일 도메인 (UI 한 화면, 함수 1~3개) | 1 | "Phase 1: 단일" |
| 여러 도메인 (서버 + 웹) | 2~3 | Phase 1 서버, Phase 2 웹 (의존: 1) |
| DB 마이그레이션 + 코드 변경 | 2~5 | Phase 1 마이그레이션, Phase 2 백엔드, Phase 3 프론트 |
| 5개 초과 예상 | — | plan 자체를 분할하거나 작업 범위 축소 (메인에 질문) |

phase 간 의존을 반드시 명시. `nl-generate` 가 의존 순서대로 실행하며, 한 phase 가 실패하면 후속 phase 는 실행하지 않는다.

## 길이 가이드

전체 plan 파일은 **80 ~ 200 줄** 권장. 200 줄 초과 시 plan 분할 또는 단계 축소.

## "이번에 안 할 것" 작성 가이드

요구사항 본문에 명시되지 않은 인접 영역을 의식적으로 떠올려 제외 항목을 만든다.

예시:
- 로그인 추가 → 안 할 것: "비밀번호 재설정", "OAuth 연동", "rate-limit"
- 차트 추가 → 안 할 것: "차트 인터랙션 (드래그/줌)", "차트 라이브러리 교체 검토"

## "트레이드오프" 작성 가이드

선택지가 갈렸던 지점만 적는다. 선택의 여지가 없었던 결정 (예: "TypeScript 사용") 은 적지 않는다.

## "Evaluate" 작성 가이드

`## 12. Evaluate ★` 섹션은 각 phase 구현 + reviewer 통과 직후 자동 실행할 외부 검증 명령 목록이다.

**작성 룰**

- 명령은 한 줄에 하나씩, 직렬 실행 순서대로 나열.
- 첫 실패 시 후속 명령은 실행되지 않는다 → 의존 순서(typecheck → lint → test → build)로 배치.
- 외부 빌드 도구가 없는 plan 은 `none` 한 단어만 기재 (빈 줄 금지).

**예시 — 채운 경우**

```
python -m mypy scripts/
python -m unittest discover tests
```

**예시 — 외부 도구 없는 경우**

```
none
```

**판정 기준** (`validate_plan.py`)

| 본문 | 판정 |
|------|------|
| 비어 있음 / 섹션 없음 | 빈 섹션 오류 |
| `none` 한 줄 | 합법 (의도적 없음) |
| 실제 명령 1줄 이상 | 합법 |
| `TBD` / `TODO` / `-` 만 | ambiguous (placeholder 금지) |
